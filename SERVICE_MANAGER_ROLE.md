# Роль Service Manager - Управление пользователями сервиса

## Описание

Роль **service-manager** позволяет назначенным пользователям управлять **пользователями и ролями конкретного сервиса**, не имея при этом административных прав внутри самого сервиса.

### Разделение ролей:

- **admin** (роль в сервисе) - административные права **внутри сервиса** (например, доступ к админ-панели referal)
- **service-manager** (роль в сервисе) - управление **пользователями и ролями через auth-service**, но без прав внутри сервиса

Это позволяет делегировать управление доступами HR-менеджерам или другим сотрудникам, не давая им полный контроль над функциональностью сервиса.

## Как это работает

### Привязка к конкретным сервисам:

Service-manager назначается **для каждого сервиса отдельно**:

```javascript
// Yu.shashkova - менеджер referal
{
  user_id: ObjectId('USER_ID'),
  service_key: 'referal',           // ← Управляет только referal
  role_name: 'service-manager',
  is_active: true
}

// Если нужно управлять несколькими сервисами - создаем несколько записей
{
  user_id: ObjectId('USER_ID'),
  service_key: 'client-service',    // ← И client-service
  role_name: 'service-manager',
  is_active: true
}
```

## Возможности service-manager

Service-manager управляет **пользователями и ролями через auth-service**, но **НЕ имеет** прав внутри самого сервиса.

### Что может service-manager:

1. **Управление пользователями сервиса** (через `/services/{serviceKey}`):
   - ✅ Добавлять новых пользователей в сервис
   - ✅ Удалять пользователей из сервиса
   - ✅ Изменять роли пользователей

2. **Управление ролями:**
   - ✅ Создавать новые роли в сервисе
   - ✅ Редактировать существующие роли
   - ✅ Удалять роли
   - ✅ Назначать разрешения ролям

3. **Просмотр информации:**
   - ✅ Видеть список пользователей сервиса
   - ✅ Просматривать роли и разрешения
   - ✅ Экспортировать данные о пользователях

### Что НЕ может service-manager:

1. ❌ Использовать административные функции **внутри сервиса** (например, админ-панель referal)
2. ❌ Создавать/удалять сервисы
3. ❌ Импортировать пользователей через Excel (только System Admin)
4. ❌ Управлять настройками auth-сервиса
5. ❌ Назначать роль GOD или admin в auth-сервисе

### Пример использования:

**Сценарий:** HR-менеджер должен добавлять сотрудников в систему referal, но не должен иметь доступ к финансовой информации и статусам сделок.

**Решение:**
- Назначить HR-менеджеру роль `service-manager` в сервисе `referal`
- HR-менеджер получает доступ к `/services/referal` (auth-service)
- Может добавлять пользователей, назначать им роли
- НО не может войти в `/referal` (сам сервис) и видеть сделки

## Как назначить роль service-manager

### Через MongoDB:

```javascript
db.user_service_roles.insertOne({
  user_id: ObjectId('USER_ID_HERE'),
  service: 'auth',
  role_name: 'service-manager',
  is_active: true,
  assigned_by: ObjectId('YOUR_ADMIN_ID'),
  assigned_at: new Date()
})
```

### Через интерфейс auth-сервиса:

1. Войдите как системный администратор (GOD или admin роль в auth)
2. Перейдите в раздел "Сервисы" → выберите сервис "auth"
3. В разделе "Пользователи сервиса" добавьте нужного пользователя
4. Назначьте ему роль "service-manager"

## Как это работает

### Архитектура:

```
┌─────────────────────────────────────────────────────────────┐
│                    Цепочка авторизации                       │
│                                                              │
│  1. System Admin (GOD/admin в auth) → Полный доступ ко всему│
│                      ↓ НЕТ                                   │
│  2. Service Manager (service-manager в auth) → Доступ к      │
│     управлению всеми сервисами                               │
│                      ↓ НЕТ                                   │
│  3. Service Admin (admin роль в конкретном сервисе) →        │
│     Доступ только к этому сервису                            │
│                      ↓ НЕТ                                   │
│  4. Access Denied                                            │
└─────────────────────────────────────────────────────────────┘
```

### Код middleware:

```go
func serviceAdminAuthRequired() gin.HandlerFunc {
    return func(c *gin.Context) {
        // ... получение пользователя ...
        
        // 1. Проверка системного администратора
        isSystemAdmin := hasAdminRole(user)
        if isSystemAdmin {
            c.Set("isSystemAdmin", true)
            c.Next()
            return
        }

        // 2. Проверка service-manager
        isServiceManager := hasServiceManagerRole(user)
        if isServiceManager {
            c.Set("isServiceManager", true)
            c.Set("isSystemAdmin", false)
            c.Next()
            return
        }

        // 3. Проверка админа конкретного сервиса
        if hasServiceAdminRole(user, serviceKey) {
            c.Set("isSystemAdmin", false)
            c.Next()
            return
        }

        // 4. Отказ в доступе
        c.HTML(http.StatusForbidden, "error.html", ...)
    }
}
```

## Пример использования

### Сценарий:
Компания хочет, чтобы HR-менеджер мог добавлять и удалять пользователей в разных сервисах (referal, client-service, etc.), но не хочет давать ему полные права системного администратора.

### Решение:
1. Создать учетную запись для HR-менеджера
2. Назначить ему роль `service-manager` в auth-сервисе
3. HR-менеджер получает доступ:
   - ✅ К управлению пользователями во ВСЕХ сервисах
   - ✅ К созданию и редактированию ролей
   - ❌ Но НЕ может удалить сам сервис
   - ❌ И НЕ может создавать новые сервисы

### Проверки безопасности:

- ✅ Service-manager не может удалить auth-сервис
- ✅ Service-manager не может создать новый сервис
- ✅ Service-manager не может назначить себе роль GOD
- ✅ Все действия логируются с указанием `assigned_by`

## Заключение

Роль **service-manager** обеспечивает безопасное делегирование административных функций. Системный администратор может назначить доверенное лицо, которое будет управлять пользователями и ролями во всех сервисах, не давая ему критических привилегий уровня системы.
